# WI-001 — Define the RCD Philosophy

## Status

Planned

## Purpose

Create the first philosophy document for Repository-Centric Development.

This document should explain why RCD exists, what problem it solves, and the principles that guide it.

## Problem

RCD needs a clear foundation before repository layout, templates, or AI skills are defined.

Without a philosophy document, later parts of the methodology may drift into project management, heavy documentation, or tool-specific workflows.

## Output

Create:

```text
spec/01-philosophy.md
```

## Scope

This Work Item includes:

- defining what RCD is
- explaining why RCD exists
- describing knowledge silos and knowledge decay
- defining the repository as the authoritative source of implementation knowledge
- documenting the four core principles
- documenting the Golden 5 Rules
- explaining what RCD is not

## Out of Scope

This Work Item does not include:

- repository layout details
- plan templates
- Work Item templates
- AI skill instructions
- project management process
- Git branching strategy

## Acceptance Criteria

- `spec/01-philosophy.md` exists.
- The document is concise enough to read in one sitting.
- The document clearly states that RCD is focused on implementation knowledge.
- The four core principles are documented.
- The Golden 5 Rules are documented.
- The document explicitly says that RCD is not a replacement for project management.
- The document avoids enterprise methodology bloat.

## Implementation Notes

Keep the language direct and practical.

Avoid terms like governance, ceremony, deliverables, velocity, stakeholder alignment, and similar project-management language unless specifically explaining what RCD is not.

The tone should suit small to medium teams building business applications.
